1. Introduction

This project aims to develop a Mobile Manipulator for use in the Innovation and Design Hub within NUS, with the system being purpose-built to assist and provide value to the staff stationed in the Hub for task alleviation. This approach is multi-disciplinary in nature, aiming at the mechanical, electrical and software design of the mobile manipulator to facilitate easier modification for tooling, a more user-friendly abstraction layer to code the robot to perform tasks and overall software integration.

1.1 Context

Robots are increasingly used to relieve labour shortages and handle repetitive tasks across sectors like warehousing, healthcare, and hospitality. This is accelerating a shift from factory-only deployment to human-centric environments, with robots taking on point-to-point delivery and routine workflows.

The team had an interest in knowing where the National University of Singapore benefit can from such a system. To explore this, the team decided to choose the Innovation and Design Hub as the testing ground.

The Hub was chosen as a location of interest as it is one of the largest workspaces within NUS with high student traffic from various disciplines to use the facilities to do discussions, fabrication or tinker.

1.2 The Innovation and Design Hub

Before starting any design of the system, the team needed to understand the space and operations within the Hub. 3 distinct areas were noted down, and these are the Tinkering Corner, Sandbox and ILounge.

1.2.1 Tinkering Corner

The Tinkering Corner is where staff and students have access to machines to do fabrication such as 3D printing and laser cutting. The space is also utilised for classes and workshops.

1.2.2 Sandbox

The Sandbox is the ‘classroom’ area of the Hub. It is frequently booked for events, classes and workshops. Students also come here to study or have discussions.

1.2.3 ILounge

Like Sandbox, this space is used for classes and discussion amongst students.

2. Problem Analysis

2.1 Stakeholders

The project involves the IDP Technical Staff, How Yee Teng (primary) and Patrick Tan Shoong Chin (secondary), as well as the IDP Technical Assistants. To gather insights, data collection through surveys and observations helped the team to understand their roles and contributions as well as understanding what problems the staff faced on the ground.

2.2 User Survey

User surveys conducted by the team were to collect information such as the taskings that the staff members need to conduct daily, the frequency of each task and their thoughts on how to improve their current workload.

Additional info from the staff mentioned that even a single-item pickup is disruptive: components and tools are scattered across the Hub, and requests often arrive while they’re busy, forcing them to pause their work or make students wait. They also noted that students leave used items piled on a table for staff to sort and restock before the 6 pm close, creating a rushed end-of-day process.

To gain further insights into the work that the staff need to handle, we observed the daily footfall of the Hub’s space for the span of a month alongside gathering historical data of other major work that the staff must handle.

2.3 Problem description

To ascertain the problems further, we drafted up a problem identification table utilising the 5W1H template.

Staff surveys and observations show a consistent strain in balancing time-critical tasks with student support. Accordingly, we framed a problem statement that captures these operational challenges faced by Hub staff.

2.4 Problem Statement

Several redundant tasks are slowing down the work of the Hub Staff and reducing efficiency in the Hub during the school semester. There is a need to streamline or automate these tasks.

2.5 Design Statement

To build a system that is mobile, to travel around the Hub’s environment, can manipulate and interact with the Hub’s objects and is autonomous, the staff do not need to act as a middleman

2.6 Value Proposition

To assist in handling tasks that cannot be handled by software changes due to the physical nature of interaction, a physical solution can be created that will be able to aid the staff in handling work.

This project will develop a robot that is under the “Mobile Manipulator” classification that has both traversal and mobile capabilities. This Mobile Manipulator will be able to travel around the Hub’s space with capabilities to interact and manipulate objects to deliver to the Staff. To control all this, the system must be autonomous to not burden the staff with another device they need to monitor constantly.

2.7 Design Specifications

Based on the derived problem statements and key problems identified by the staff, the design specifications were prepared. Given our limited manpower, budget of $1600 SGD and time for development of less than a year, we decided to scope down to 3 distinct tasks that the system must achieve that assists with the staff.

1. The system must be able to perform tool collection

2. The system must be capable of point-to-point delivery

3. To supplement controls to the system, a user interface must be created

2.7.1 Task 1: Tool Collection

For task 1, the robot is to grab tools from the pegboard, put it into an onboard basket and deliver it to a table. End-effector system with tool changer capabilities enables picking tools of different geometries.

2.7.2 Task 2: Point - to - point delivery

For task 2, the robot is to travel to the storeroom where several components are located to return items that the students have borrowed throughout the day. The onboard basket can be detached from the system to return items as a set.

2.7.3 Task 3: User Interface

For task 3, for the staff to control and monitor signs onboard for the system to function, a user interface is to create that can link commands from the user to the relevant commands for the hardware to perform the task.

2.8.0 Concept Development

To evaluate the ideal concepts for development, we cross compared various concepts that fulfil the tasks across various metrics such as hardware complexity, cost, scalability etc. At the end, the traditional mobile manipulator is the ideal choice as there is a high degree of flexibility in changing out end-effectors for the system, it can adapt to the environment and is statically stable even during power loss or in emergency scenarios.

2.9.0 Subsystem Identification

Based on the taskings that the robot must accomplish, we have identified the key subsystems that need to be developed to bring the system to a minimum viable product.

3. Mechanical Subsystem - End-Effector (Jason)

To design the End Effectors for the system, design constraints must be established in terms of the actuation mode of choice, overall maximum size and the maximum mass before further analysis on the approach can be done.

3.1 Actuation mode

To narrow down the ideal Actuation mode for the system, we evaluated the main types of actuation methods against various criteria for the system through the Concept Screening technique.

Based on the limitations of the payload of the AMR unit and space constraints within the body to fit all the components, the Hardware Requirements section was listed with higher weightage. Based on the concept screening and understanding of our system, we decided to choose the electric actuation mode for the design of the end-effector.

3.2 Size and Mass Constraints

The size and mass constraints of the system are decided by the following factors:

Cobot End Effector Payload Capacity: 5kg

Ideal end effector centre of gravity: Within 135mm from the end connection of J6, max width of 200mm.

Allocated space on the robot structure: 600mm x 135mm x 200mm

3.3 End Effector Analysis (Task 1: Tool pick and place)

Based on the task requirements specified, one of the end effectors would need to interact with the various tools specified on the pegboard. To get an understanding on the size and mass of the tools that the end effector must grab, the team measured and obtained the relevant data to fix our design requirements.

3.3.1 Design challenges

Between the various tools, the clamping interface must securely hold varying non-linear surfaces, with material contact surfaces varying across tools (plastic, rubber, steel), each having a different coefficient of friction which impacts the clamping force.

3.4 End Effector Design

With the design constraints known, design specifications for both the gripper and tool changer unit are specified.

3.5 Grippers

To choose the ideal gripper mechanisms, the team used a concept screening chart to evaluate different designs against key metrics.

Based on the metric scoring, the rack and pinion and lead screw approach were both chosen to further prototype and explore.

To avoid excessive contact forces onto a singular contact point, the team decided to investigate compliant gripper fingers. A common design used is the Festo Fin Ray gripper finger design. The Festo Fin Ray gripper uses a flexible, fin-like triangular skeleton with transverse ribs that passively bends around objects, so contact forces redirect into closing forces instead of peeling them away. This geometry gives gentle, adaptive gripping of irregular or fragile parts, high tolerance to misalignment, and simpler actuation/control compared with rigid parallel jaws.

3.6 Tool Changers

As the system will be interacting with various tools, a singular gripper design may be unable to carry or support all types of objects. As a result, the team aims to implement a tool changing system onboard the Cobot to facilitate end-effector changes. Due to the actuation mode being limited to being electric, the team narrowed down 3 concepts that can be explored and evaluated those against metrics.

Due to the limited space and mass constraints on the system, the Compactness/Weight criterion was weighted highly. Based on the scoring, the Permanent Electromagnet was chosen to be the concept to be explored further.

3.7 Prototypes

3.7.1 Gripper

Before analysis can be done, an approximate normal force acting on the gripper jaw was established at 10N based on supporting a load of 15N with rubber contact against metal at a coefficient of friction of 0.69.

To test the Fin Ray design, an inspired gripper was designed and simulated it in SolidWorks via Nonlinear analysis, using TPU-85A as the material. The goal of the analysis was to locate the ideal gripping section that will allow the gripper to wrap around the object.

To test the Rack and Pinion concept with the Fin Ray gripper fingers, I built a 3D-printed prototype using PLA with TPU-85A for the fingers to test the deformation and gripping strength.

Since this is a prototype, I designed it with parts that facilitate easy swapping of finger geometries to test clamping force. Initial tests showed favourable results in carrying objects up to 900g. Further test will be conducted to validate different gripper profiles and geometries.

3.7.2 Tool Changer

To test the permanent electromagnet concept, calculations were performed on the type of electromagnet used, the corresponding magnetic material to maximise the magnetic attraction between the magnet and electromagnet.

The electromagnet use was a permanent electromagnet with a 24V demagnetisation input with 30kg rated strength. For the magnetic material, S275 Ferritic Steel was chosen with 6mm thickness to maximise the magnetic field.

Current tests indicate that the magnet can sustain 10kgs of load for a sustained period. Future works into the next semester will focus on creating the tool changer system with the mechanical body and electrical connections.

5. Software Subsystems (Jason)

5.1. Software Strategy

To ensure completion of the 3 main tasks assignment, an understanding must be done on how the stakeholders would interact with the system and get it to completion.

A user sequence diagram was created to showcase the generic workflow of the system by the user.

The Robot Software System consists of:

- AMR to Arm Communication

- Task planning and motion control

- Manipulator firmware and control

The Robot Hardware System consists of:

- AMR

- Cobot

- End Effector System

A robot sequence diagram was also created to show the motions of the robot upon receiving a command.

5.2 AMR to Cobot Communication

To allow the AMR and the Cobot to establish communication, the team decided to use the ROS2 Framework as developing a custom software to fully integrate both systems would be challenging given the closed-source nature of the AMR and Cobot.

5.3 Task planning and motion control

To make the system work as a unified robot system, the robot software must know how to establish the parsing of commands from the user to clear instructions to the hardware.

A software container diagram using the C4 model notation was used to show the communication between nodes to create the Robot Software System.

5.3.1 Safety Interfacing

Between the AMR and Cobot, there are several safety features in place to protect the user in the event of a collision. To add on to monitoring these peripherals, the software system will send movement commands only if the AMR is stationary and vice versa. A programmed protective bubble governed by an ellipsoid equation will be activated around the Cobot during operation to ensure incorrect trajectory calculations/movements will be terminated.

5.3.2 Cobot Trajectory Planner

For the Cobot to reach a desired position in space, a positional command made of Vectors and Quaternions must be sent to establish both relative position and orientation, which is called a “frame”. To move between 2 frames with a specified trajectory and constraints would require a trajectory planner.

Since developing a planner was not in the scope of the project, the team opted to use planners available in the MoveIt2 Library with ROS2 to fit the Cobot. There are different planners, each with their own advantages. 3 planners were chosen based on the Cobot ROS2 driver compatibility. These are OMPL (Open Motion Planning Library), CHOMP (Covariant Hamiltonian Optimization for Motion Planning) and PILZ (PILZ Industrial Motion Planner). We cross-compared the planners with our use case based on metrics seen in the table below.

For scoring, we scored it from 1 – 3, with 3 being ideal. Due to the conditions that the robot system must interact with, planners with unpredictable behaviour or have probability-based planning were rated lower. Planners with high computational requirement were also rated low. PILZ ended up being the most ideal planner for our use case as it chooses the shortest path while following velocity constraints with high repeatability and predictability.

The planner was defined for use in the test_rv5as.py node such that any input commands will move the robot system accordingly.

5.3.3 Waypoint finding

For the AMR to reach a desired waypoint, an API call to the Waypoints API will allow for requesting a specific goal point for the robot to travel too that can also be saved via another API call to the Goal API.

5.4: End Effector Firmware and Control

To control the end-effector, the current proposed plan is to utilise the ROS2 system as the higher-level control whilst a Microcontroller will act as the lower-level control that deals with the motors, sensors and other peripherals on the end-effector unit.

5.5: Graphical User Interface (GUI)

Based on Task 3, a user interface is to be created to allow the stakeholders to send requests to the system to perform tasks.

Although both the AMR and Cobot have their own respective GUIs for control, to create a unified robot system would mean a singular user interface that allows for control of both systems.

Our final GUI vision is a single, unified interface that controls both the Cobot and the AMR. Built for staff and makers, it enables task execution without any coding, using clear workflows and teach-by-demonstration tools. A dedicated teaching mode streamlines adding goals and waypoints, making everyday operation fast and intuitive.
